# 食光记前端技术方案（微信原生 + Taro）

## 1. 文档说明

### 1.1 文档目标

本文档基于以下材料，输出“食光记”小程序前端技术方案，作为后续前端开发、联调、测试与迭代的指导文档：

- `docs/需求文档v2.md`
- `stitch_prd/` 页面原型与 `heritage_hearth/DESIGN.md`
- `docs/backend/后端技术方案.md`
- `docs/backend/API 接口清单.md`
- `docs/backend/数据库详细设计.md`

本文档重点回答以下问题：

- 微信原生 + Taro 的前端工程如何落地。
- 页面、路由、分包、状态和接口应该如何组织。
- 前端与后端接口边界如何约定，哪些逻辑必须后置到服务端。
- 如何在保证 MVP 开发效率的同时，为随机点菜、家庭协作等后续能力预留空间。

### 1.2 适用范围

- 当前阶段：微信小程序 MVP。
- 渲染目标：微信小程序优先，保留 Taro 跨端扩展能力，但不以 H5/App 同步交付为第一目标。
- 协作边界：本文档只覆盖前端实现方案，不替代视觉设计稿、接口文档和测试用例文档。

### 1.3 设计原则

- 微信优先：优先保证微信小程序端体验、兼容性和原生能力接入，而不是为了“跨端”牺牲可用性。
- Taro 管工程，原生能力做体验：工程层使用 Taro，交互和端能力优先调用微信小程序能力。
- 业务规则后置：版本号递增、差异摘要、购物清单聚合、随机点菜规则均由后端负责，前端只负责输入、展示和状态驱动。
- 页面闭环优先：优先打通“菜谱 -> 版本 -> 时光记录 -> 点菜 -> 购物清单”核心主线。
- 渐进增强：复杂动画、自定义手势、家庭协作、分享图高级样式等能力放在主链路稳定后迭代。

## 2. 产品与原型边界

### 2.1 当前 MVP 功能范围

按 PRD 与后端设计，前端 MVP 以以下能力为主：

- 首页：本周菜单入口、最近时光流。
- 菜谱库：搜索、分类、标签筛选、列表展示。
- 菜谱创建：创建菜谱并自动生成 V1。
- 菜谱详情：做法、版本、时光轴三大视图。
- 新建版本：基于历史版本复制草稿并创建新版本。
- 时光记录：文字、日期、图片、参与人、评分。
- 点菜台：周视图、手动加入菜品、调整顺序、删除。
- 购物清单：生成、勾选、备注、复制文本。
- 我的：分类管理、标签管理、设置入口。

- 家庭协作与成员权限页面
- PDF 导出、打印手账
- 全局时光轴独立页
- 复杂拖拽排序
- 完整离线同步
- 重型图像编辑能力

## 3. 技术选型

### 3.1 推荐栈

| 层级 | 方案 | 说明 |
| --- | --- | --- |
| 框架 | Taro + React + TypeScript | 使用 Taro 统一工程与构建，React 便于组件化和状态组织 |
| 端能力 | 微信小程序原生 API，经 Taro 封装调用 | 图片选择、上传、震动、剪贴板、分享等直接走原生能力 |
| 样式方案 | SCSS Modules + 设计 Token | 保持小程序样式可控，避免运行时样式工具增加复杂度 |
| 服务端状态 | TanStack Query | 统一列表缓存、请求去重、失效刷新和分页 |
| 轻量全局状态 | Zustand | 只承载会话、筛选草稿、局部跨页状态 |
| 表单校验 | Zod + 页面本地状态 | 规则与接口字段统一，避免脏数据提交 |
| 网络请求 | `Taro.request` 二次封装 | 统一鉴权、错误码、时间格式、日志字段 |
| 时间处理 | Day.js 或等价轻量库 | 统一周起始日、日期格式和展示 |
| 代码规范 | ESLint + Prettier + Stylelint | 保持多人协作后仍易维护 |
| 包管理 | pnpm | 降低依赖安装成本，便于 monorepo/后续扩展 |

### 3.2 不建议的方案

- 不建议首期引入重型 UI 组件库。
  原型风格较强，现成 UI 库很难贴合，反而会增加重写成本。
- 不建议首期使用 CSS-in-JS。
  小程序环境下调试和性能收益有限，SCSS Modules 更稳定。
- 不建议首期引入离线数据库。
  MVP 业务是在线 CRUD，复杂同步机制收益低。
- 不建议首期启用复杂跨端兜底逻辑。
  先把微信端体验做稳，再决定是否真正扩 H5/App。

### 3.3 关于“微信原生 + Taro”的落地理解

本项目不采用“纯原生小程序散页开发”，也不采用“完全 Web 化页面思维”。建议采用如下折中方案：

- 页面、路由、构建、组件树用 Taro 管理。
- 视觉和交互尽量贴近微信原生页面习惯。
- 图片、震动、分享、上传、剪贴板等直接调用原生能力。
- 复杂动画和端差异能力通过“组件封装 + 原生 API”方式实现，不把业务代码写死在页面里。

## 4. 总体架构设计

### 4.1 前端分层

建议采用以下分层：

```text
Page
  -> Feature Hooks
    -> Query / Store
      -> Service
        -> Request Client
          -> Backend API
```

各层职责如下：

| 层 | 职责 |
| --- | --- |
| Page | 页面布局、页面生命周期、交互编排 |
| Feature Hooks | 业务行为封装，如“创建版本”“生成购物清单” |
| Query / Store | 服务端缓存、本地 UI 状态、页面草稿 |
| Service | 调接口、DTO 转换、上传链路封装 |
| Request Client | 统一请求头、错误处理、日志埋点字段注入 |

### 4.2 目录结构建议

建议单独建立小程序工程目录，例如：

```text
miniapp/
  config/
    dev.ts
    prod.ts
    index.ts
  src/
    app.tsx
    app.config.ts
    pages/
      home/
      recipe-library/
      meal-planner/
      my/
    packages/
      recipe/
        detail/
        edit/
        version-create/
        version-compare/
        moment-edit/
      planner/
        random-pick/
        shopping-list/
      profile/
        category-manage/
        tag-manage/
        settings/
    components/
      base/
      business/
    features/
      auth/
      taxonomy/
      recipe/
      moment/
      meal-plan/
      shopping/
      random/
    services/
      request/
      modules/
      types/
    store/
    hooks/
    utils/
    constants/
    styles/
      tokens.scss
      mixins.scss
      reset.scss
```

### 4.3 模块化约定

`features/` 下按业务域组织，而不是按“hooks/components/utils”大一统组织。每个业务域建议包含：

```text
features/recipe/
  api.ts
  query.ts
  store.ts
  mapper.ts
  types.ts
  hooks.ts
  components/
```

好处：

- 更贴近后端模块边界。
- 方便后续拆人或拆分包。
- 页面依赖关系更清晰，减少“公共目录垃圾化”。

## 5. 路由与分包方案

### 5.1 主导航建议

MVP 建议保留 4 个一级 tab：

- 首页
- 菜谱库
- 点菜台
- 我的

原因：

- 与 PRD 信息架构一致。
- 用户主路径清晰。
- 随机点菜和购物清单都属于“点菜台”的派生能力，不应再占一级导航。

### 5.2 页面路由建议

| 路由 | 页面 | 分包 | 备注 |
| --- | --- | --- | --- |
| `/pages/home/index` | 首页 | 主包 | tab |
| `/pages/recipe-library/index` | 菜谱库 | 主包 | tab |
| `/pages/meal-planner/index` | 点菜台 | 主包 | tab |
| `/pages/my/index` | 我的 | 主包 | tab |
| `/packages/recipe/detail/index` | 菜谱详情 | `recipe` | 核心二级页 |
| `/packages/recipe/edit/index` | 新建菜谱 / 编辑基础信息 | `recipe` | 创建时含 V1 表单 |
| `/packages/recipe/version-create/index` | 新建版本 | `recipe` | 从指定版本复制草稿 |
| `/packages/recipe/version-compare/index` | 版本对比 | `recipe` | 根据版本 ID 对比 |
| `/packages/recipe/moment-edit/index` | 记一笔 | `recipe` | 支持新建与编辑 |
| `/packages/planner/random-pick/index` | 随机点菜 | `planner` | 一期可灰度 |
| `/packages/planner/shopping-list/index` | 购物清单 | `planner` | 与周计划绑定 |
| `/packages/profile/category-manage/index` | 分类管理 | `profile` | 列表 + 表单抽屉 |
| `/packages/profile/tag-manage/index` | 标签管理 | `profile` | 列表 + 表单抽屉 |
| `/packages/profile/settings/index` | 设置 | `profile` | 先放静态和基础设置 |

### 5.3 分包原则

分包应遵循以下规则：

- 主包只放 tab 页和必要公共组件，保证首页启动快。
- 二级业务页按“菜谱域 / 计划域 / 我的域”拆分。
- 路由参数只传 `id`、`weekStartDate` 等轻量值，不传大对象。
- 菜谱详情、版本对比、时光编辑共用菜谱分包，减少重复代码。

### 5.4 关于 tabBar 的实现建议

MVP 建议优先使用原生 `tabBar` 配置，而不是首版就做自定义 tabBar。

原因：

- 成本更低，稳定性更好。
- 当前主要差异化在内容卡片和页面排版，不在底部导航动效。
- 原型风格可通过页面背景、图标和文案统一，而不一定依赖自定义 tabBar。

如果二期需要更强的品牌一致性，再评估升级为自定义 tabBar。

## 6. 页面与业务实现方案

### 6.1 首页

页面职责：

- 展示“本周已规划 X 道菜”卡片。
- 展示最新时光动态流。
- 提供“新建菜谱”“进入随机点菜”的快捷入口。

前端实现重点：

- 页面首屏优先请求本周菜单摘要和最新时光流。
- 最新时光流支持分页加载，不一次性加载所有记录。
- 点击时光卡片跳转菜谱详情，并可带上 `momentId` 作为定位参数。
- 空状态要给出明确 CTA，如“去创建第一道菜”。

### 6.2 菜谱库

页面职责：

- 分类筛选
- 标签筛选
- 关键词搜索
- 菜谱卡片列表

前端实现重点：

- 分类列表、标签列表进入页面时并行请求。
- 搜索框做 300ms 防抖，减少无意义请求。
- 筛选条件变化时重置分页。
- 网格/列表视图切换只保存在本地，不需要服务端介入。
- 卡片展示字段与后端列表 DTO 保持一致，不自行拼“版本差异”。

### 6.3 新建菜谱 / 编辑基础信息

建议把“新建菜谱”和“编辑基础信息”合并为一个表单页，但行为分开：

- 创建模式：提交 `POST /recipes`，包含 V1 内容。
- 编辑模式：提交 `PATCH /recipes/:id`，只更新菜谱基础信息，不直接改历史版本。

表单建议分区：

- 基础信息：菜谱名、封面、分类、标签
- 当前版本内容：版本别名、主料、步骤、小贴士

交互建议：

- 分类和标签支持“选择现有 + 临时新建”。
- 步骤采用可增删的数组表单。
- 首期不强做复杂拖拽排序，使用“上移/下移”按钮即可。

### 6.4 菜谱详情

页面采用三段式结构：

- 顶部信息区：封面、菜名、分类、标签、当前版本
- 操作区：记一笔 / 新建版本 / 加入点菜
- 内容区：做法 / 版本 / 时光轴

实现建议：

- Tab 内容按需加载，避免详情页初次渲染请求过多。
- 做法页读取 `GET /recipes/:id`。
- 版本页读取 `GET /recipes/:id/versions`。
- 时光轴读取 `GET /recipes/:id/moments`。
- 若存在 `momentId` 参数，则在时光轴加载完成后滚动定位。

### 6.5 新建版本

新建版本页是 MVP 核心页面之一，建议独立页实现，不与菜谱详情内嵌表单混用。

处理流程：

1. 通过路由参数接收 `recipeId` 与 `sourceVersionId`。
2. 先请求源版本详情，作为草稿默认值。
3. 用户修改后提交创建版本接口。
4. 创建成功后返回详情页，并提示是否切换到版本页查看。

注意事项：

- 版本号由后端生成，前端不做本地自增。
- 差异摘要由后端返回，前端只展示 `diffSummaryText` 与结构化标签。
- 若从当前版本发起创建，页面文案显示“基于当前版本创建”。

### 6.6 版本对比

版本对比页只承担“可读展示”，不承担比对计算。

页面建议包含：

- 头部版本信息
- 差异摘要卡片
- 分类/标签/主料/步骤数变化
- 两侧版本完整内容展开区
- “设为当前版本”操作按钮

接口策略：

- 概览走 `GET /recipes/:id/compare`
- 若需要完整内容，再并行请求两个版本详情

### 6.7 时光记录

时光记录页应支持“新建”和“编辑”双模式。

核心能力：

- 选择关联菜谱与版本
- 选择日期
- 输入内容
- 添加参与人
- 评分
- 上传最多 9 张图片

实现建议：

- 图片上传采用“两段式上传”。
- 图片选择走原生媒体能力，提交表单时只上传 `imageAssetIds`。
- 若从菜谱详情进入，默认带入 `recipeId` 和当前 `recipeVersionId`。
- 保存成功后回跳详情页并刷新时光轴。

### 6.8 点菜台

点菜台页面以“周视图 + 当日菜单项列表”为主。

实现建议：

- 页面路由以 `weekStartDate` 为核心参数。
- 默认打开当前周，支持切换上一周 / 下一周。
- 添加菜品时使用弹层或跳转选择页，而不是直接在周格子里塞复杂搜索。
- 菜单项必须绑定具体版本。
- 排序首期使用“上移/下移”或轻量重排，不强依赖拖拽。

### 6.9 随机点菜

随机点菜页属于体验增强页，但建议前端结构在第一版就预留。

前端职责：

- 维护筛选条件表单。
- 调用创建随机 session 接口。
- 用前端动画包装后端结果，而不是前端自行随机。
- 接收“就它了 / 再来一次 / 跳过 / 换版本”等操作。

关键原则：

- 候选池与命中逻辑以后端为准。
- 前端动画只负责氛围表达，最终结果以接口返回为准。
- 震动反馈、减速效果可以走原生 API + CSS 动画实现。

### 6.10 购物清单

购物清单页与周菜单强绑定，建议从点菜台进入。

实现建议：

- 若当前周无已生成清单，先引导生成。
- 详情页分“原料 / 调料”两个分栏展示。
- 勾选状态更新做乐观更新，失败时回滚并提示。
- “复制文本”直接调用后端文本生成接口，前端只负责放入剪贴板。
- 分享图能力先预留按钮，若后端异步生成则展示任务处理中状态。

### 6.11 我的 / 分类 / 标签 / 设置

“我的”页是二级管理入口，不建议塞入过多复杂业务。

建议结构：

- 顶部个人信息 / 当前家庭摘要
- 分类管理入口
- 标签管理入口
- 设置入口

分类和标签管理页建议使用同一套列表管理模式：

- 列表展示
- 新增/编辑弹层
- 删除二次确认
- 分类支持重排

## 7. 状态管理方案

### 7.1 状态分类

前端状态按三类处理：

| 类型 | 示例 | 方案 |
| --- | --- | --- |
| 服务端状态 | 菜谱列表、版本列表、时光流、周菜单 | TanStack Query |
| 全局 UI 状态 | 当前会话、筛选草稿、选中的周起始日 | Zustand |
| 页面表单状态 | 新建菜谱草稿、版本草稿、时光记录草稿 | 页面本地状态 |

### 7.2 Query Key 约定

建议统一 Query Key，便于后续失效控制：

```text
['session']
['categories']
['tags']
['recipes', filters]
['recipe-detail', recipeId]
['recipe-versions', recipeId]
['recipe-compare', recipeId, baseVersion, targetVersion]
['recipe-moments', recipeId, pagination]
['latest-moments', params]
['meal-plan', weekStartDate]
['shopping-list', shoppingListId]
['random-session', sessionId]
```

### 7.3 失效刷新规则

| 操作 | 需要刷新 |
| --- | --- |
| 创建菜谱 | 菜谱列表、首页快捷统计 |
| 新建版本 | 菜谱详情、版本列表、菜谱列表 |
| 记录时光 | 菜谱详情时光轴、首页最新时光、菜谱列表封面摘要 |
| 点菜 | 当前周菜单、首页本周统计 |
| 生成购物清单 | 当前周菜单状态、购物清单详情 |
| 分类/标签变更 | 分类列表、标签列表、菜谱编辑页筛选数据 |

### 7.4 草稿保存建议

以下页面可以做本地草稿缓存，但仅缓存未提交内容，不做正式离线同步：

- 新建菜谱
- 新建版本
- 时光记录

缓存建议：

- 仅在本地存储轻量文本与字段，不缓存大图二进制。
- 草稿带时间戳，超过 7 天自动清理。
- 草稿恢复需显式询问用户，避免误覆盖。

## 8. 接口与数据契约方案

### 8.1 统一响应适配

后端已定义统一响应结构，前端请求层应统一适配：

```ts
type ApiSuccess<T> = {
  success: true
  data: T
  requestId: string
}

type ApiFailure = {
  success: false
  error: {
    code: string
    message: string
    details?: unknown
  }
  requestId: string
}
```

请求层统一处理：

- `success === false` 时抛出业务错误
- 根据错误码映射 toast 文案
- 打印 `requestId` 便于联调排查

### 8.2 服务模块建议

```text
services/modules/
  auth.ts
  taxonomy.ts
  recipes.ts
  moments.ts
  meal-plans.ts
  random-picks.ts
  shopping-lists.ts
  media.ts
```

每个模块只负责：

- 请求参数定义
- DTO 到 ViewModel 的轻量转换
- 上传链路或分页处理封装

不负责：

- 页面交互逻辑
- 弹窗提示
- 复杂状态拼接

### 8.3 关键接口约定

前端实现时重点遵循以下边界：

- 菜谱列表只使用后端返回的当前版本摘要，不前端自行拼接当前版本信息。
- 版本对比只展示后端返回的 `summaryText / summaryJson`。
- 购物清单列表只展示后端聚合后的快照数据，不前端重复做去重。
- 随机点菜只消费 session 结果，不在前端复刻筛选规则。
- 日期字段统一传 `YYYY-MM-DD`。

### 8.4 鉴权实现建议

前后端统一采用“微信 `code` 登录 + 后端签发 `accessToken / refreshToken`”方案，不使用 Cookie Session 作为小程序主会话机制。

推荐流程：

1. 小程序端调用 `wx.login()` 获取临时 `code`
2. 前端调用 `POST /api/v1/auth/wechat-login`
3. 后端调用微信 `code2Session` 获取 `openid`
4. 后端查找或创建当前业务用户，并签发：
   - `accessToken`：短期有效，用于受保护接口
   - `refreshToken`：较长期有效，用于刷新 `accessToken`
5. 前端把 token 存本地，并在请求头统一注入 `Authorization: Bearer <accessToken>`
6. `accessToken` 过期后，前端调用刷新接口换新 token

前端请求层要求：

- 不依赖浏览器自动带 Cookie。
- 统一处理 `401 -> refresh -> 重放请求`。
- `refreshToken` 刷新失败后，清理本地会话并回到登录初始化流程。

结论：

- 小程序端鉴权以 token 为准。
- 后端若未来为 Web 管理端保留 Cookie/Auth.js 能力，可以并存，但不能作为当前小程序主链路依赖。

### 8.5 图片上传链路

时光记录和封面上传统一走两段式：

1. 调 `POST /media/upload-token`
2. 使用 `Taro.uploadFile` 或等价原生能力上传二进制
3. 调 `POST /media/complete` 登记资源
4. 拿到 `assetId` 后提交业务表单

前端需要统一封装：

- 文件大小和格式校验
- 上传进度
- 单张失败重试
- 超过 9 张时的拦截提示

## 9. 设计系统与样式落地

### 9.1 视觉基调

`stitch_prd/heritage_hearth/DESIGN.md` 已明确本项目不是通用工具型 UI，而是“家庭数字手账”风格。前端落地时要保持以下核心特征：

- 温暖底色，而非纯白冷感页面
- 以卡片层次和色块区分区域，而不是依赖细边框
- 标题更具编辑感，正文保持清晰易读
- 重点操作保留“手账感”而不是通用运营风

### 9.2 Token 设计建议

建议将设计 Token 抽成 `styles/tokens.scss` 与 `constants/theme.ts` 两份：

- `SCSS`：用于页面和组件样式
- `TS`：用于 JS 逻辑、图表、状态色映射

建议至少沉淀：

- 颜色：`primary / secondary / tertiary / surface / surface-container-*`
- 字号：`display / headline / title / body / label`
- 间距：`4 / 8 / 12 / 16 / 24 / 32 / 40`
- 圆角：`sm / md / lg / pill`
- 阴影：仅保留 2 到 3 档轻量阴影

### 9.3 字体策略

原型中使用了在线字体，但小程序正式环境不应直接依赖外网字体资源。

推荐策略：

- 标题字体：优先使用本地可控字体资源，未落地前使用系统衬线兜底。
- 正文字体：使用系统无衬线字体兜底。
- 若正式版确需品牌字体，可评估通过 `loadFontFace` 动态加载，但不应阻塞 MVP 交付。

### 9.4 组件清单建议

建议优先沉淀以下通用组件：

- `PageContainer`
- `TopNavBar`
- `StickyActionBar`
- `RecipeCard`
- `VersionBadge`
- `DiffSummaryCard`
- `TimelineCard`
- `FilterChips`
- `SearchBar`
- `UploadGrid`
- `RatingSelector`
- `WeekBoard`
- `ShoppingItemCard`
- `EmptyState`
- `ConfirmDialog`

原则：

- 先做业务组件，再做过度抽象。
- 同类页面的交互骨架尽量统一，减少心智切换。

## 10. 性能与兼容性方案

### 10.1 首屏与包体策略

- 首页、菜谱库、点菜台、我的放主包。
- 详情、对比、表单页放分包。
- 图片采用缩略图优先展示，详情页再加载原图。
- 列表分页每次控制在合理条数，避免首次加载过重。

### 10.2 请求性能

- 首页采用并行请求，但要控制在 2 到 3 个核心请求内。
- 搜索和筛选使用防抖。
- 长列表使用分页或“加载更多”，不做无限首屏渲染。
- 对于分类、标签等低频变更数据，允许短期缓存。

### 10.3 渲染性能

- 页面中避免深层匿名函数和大对象透传。
- 卡片列表组件尽量纯展示化，减少无关重渲染。
- 大表单拆为多个子组件，避免输入时整页刷新。
- 不在路由参数中传输大型 JSON。

### 10.4 原生能力使用建议

| 场景 | 建议能力 |
| --- | --- |
| 图片选择 | `chooseMedia` |
| 图片上传 | `uploadFile` |
| 剪贴板 | `setClipboardData` |
| 随机点菜震动反馈 | `vibrateShort` |
| 图片预览 | `previewImage` |
| 分享图保存 | `saveImageToPhotosAlbum`，待后端分享图能力稳定后接入 |

### 10.5 关于 Skyline 的建议

首期不建议把 Skyline 作为强依赖方案。

原因：

- 当前页面复杂度尚未高到必须依赖新渲染能力。
- MVP 更需要稳定联调与可维护性。
- 后续如果首页卡片流或复杂动画存在明显瓶颈，再专项评估。

## 11. 工程规范建议

### 11.1 命名规范

- 页面目录：小写中划线，如 `recipe-library`
- 组件目录：大驼峰，如 `RecipeCard`
- Hook：`useXxx`
- Store：`useXxxStore`
- Query Hook：`useXxxQuery`、`useXxxMutation`
- 类型：`RecipeSummary`、`MomentItem` 等领域命名

### 11.2 代码组织规范

- 页面只做页面逻辑，不直接写请求。
- 所有请求统一从 `services/modules` 出口进入。
- 业务常量放 `constants/`，不要散落页面内。
- DTO 和 ViewModel 分离，避免后端字段变更直接冲击页面。

### 11.3 环境变量建议

建议至少包含：

- `TARO_APP_API_BASE_URL`
- `TARO_APP_UPLOAD_BASE_URL`
- `TARO_APP_ENABLE_RANDOM_PICK`
- `TARO_APP_ENABLE_MOCK`
- `TARO_APP_LOG_LEVEL`

### 11.4 错误与提示规范

- 用户错误优先给出可操作提示。
- 服务错误统一 toast + 错误页/空态兜底。
- 删除、切换版本、清空已购等高风险操作必须二次确认。
- 表单保存成功后给出明确反馈，并说明下一步去向。

## 17. 合规与适配要求（微信最新，必满足）
### 17.1 隐私与合规设计（微信审核必过）
- 隐私政策弹窗：首次打开小程序时弹出，居中展示（宽度300rpx，高度400rpx），含“标题（24rpx）+ 简要说明（18rpx，≤50字）+ 同意/不同意按钮”，不同意则退出小程序；
- 授权规范：所有用户信息授权（头像、昵称），均需用户主动点击按钮触发，同一授权项拒绝后不再主动弹窗（仅提供手动授权入口）；
- 敏感信息处理：头像昵称仅在个人中心展示，导出记录中的账号信息按“昵称+脱敏标识”展示，输入框如涉及来源链接、备注等敏感信息时不自动公开展示；
- 备案信息展示：备案号展示在首页底部（文字16rpx，中性色3），点击跳转备案信息静态页（含备案号、备案主体、查询入口）。

### 17.2 无障碍适配（微信审核加分项）
- 语义化标签：所有交互组件使用微信原生语义化标签（button、form、input），不自定义非语义化标签；
- aria-label属性：图标、无文字按钮添加aria-label属性（如搜索清空按钮添加“清空搜索”，关闭按钮添加“关闭弹窗”）；
- 可点击区域：所有可点击组件点击热区≥50rpx×50rpx，间距≥8rpx，避免误触；
- 文字可放大：支持微信系统字体放大200%，放大后不重叠、不溢出，布局保持正常。

### 17.3 兼容性适配（量化覆盖范围）
- 微信版本：支持微信最新3个版本+历史兼容版（覆盖≥98%用户）；
- 设备适配：3.5-7.2英寸屏幕（含折叠屏、刘海屏、全面屏），竖屏为主，平板横屏适配（核心内容居中，左右留白）；
- 系统适配：iOS 15+/Android 10+，无卡顿、闪退、布局错乱（崩溃率≤0.1%）；
- 性能适配：页面切换流畅（无卡顿时长≥100ms），核心功能操作响应≤800ms，弱网环境核心功能可正常使用
