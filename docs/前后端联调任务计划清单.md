# 食光记前后端联调任务计划清单

## 1. 适用范围

本文档基于 2026-04-09 当前代码状态制定，目标是在：

- 后端已完成阶段 3：鉴权、taxonomy、recipes、recipe versions 主链路可用。
- 前端已完成阶段 2：登录、分类/标签、菜谱库、菜谱创建/编辑、详情、版本创建/对比页面可用。

前提下，给出一份最适合当前项目节奏的联调顺序与任务清单，优先确保最短业务闭环尽快稳定。

## 2. 当前判断

### 2.1 已具备联调条件的能力域

- Auth：`wechat-login / refresh / session / logout`
- Taxonomy：`categories / tags`
- Recipes：列表、详情、创建、更新、删除
- Recipe Versions：列表、详情、创建、版本对比、设为当前版本

### 2.2 暂不适合进入真接口联调的能力域

- Moments / Media：前端仍是占位页，后端阶段 4 也未完成
- Menu Plans：前端只有 mock `current-week`，后端未提供对应接口
- Shopping List：前端占位，后端未完成
- Random Pick：前端占位，后端未完成

### 2.3 当前最关键的联调策略

当前最合适的策略不是“全量切真接口”，而是：

1. 先完成接口契约对齐，避免页面一接真接口就因为字段不一致而大面积返工。
2. 再按 Auth -> Taxonomy -> Recipes -> Versions 的顺序联调。
3. 每联通一个页面，就立即关闭该页面对应的 mock 依赖，避免双轨长期存在。
4. 时光记录、点菜台、购物清单、随机点菜继续留在下一阶段，不混入本轮计划。

## 3. 已识别的联调风险

这些问题建议作为联调任务 0 优先处理，否则后续页面联调会反复卡住。

### 3.1 菜谱版本列表返回结构不一致

- 前端当前 `recipeService.getRecipeVersions()` 期望直接返回数组，见 [recipe.ts](/Users/xychen/Documents/MyCode/menu-time/miniapp/src/services/modules/recipe.ts)。
- 后端实际定义为分页结果 `PageResult<RecipeVersionListItemDto>`，见 [recipes.types.ts](/Users/xychen/Documents/MyCode/menu-time/backend/src/server/modules/recipes/recipes.types.ts)。

影响：

- 版本页读取真接口后会直接出现数据结构不匹配。

### 3.2 版本对比接口参数语义不一致

- 前端当前 `compareRecipeVersions(recipeId, baseVersionId, targetVersionId)` 把 `base`、`target` 作为字符串版本 ID 传给 `/compare`，见 [recipe.ts](/Users/xychen/Documents/MyCode/menu-time/miniapp/src/services/modules/recipe.ts)。
- 后端当前 `/api/v1/recipes/:id/compare` 要求的是版本号数字 `base`、`target`，见 [recipes.schema.ts](/Users/xychen/Documents/MyCode/menu-time/backend/src/server/modules/recipes/recipes.schema.ts)。

影响：

- 版本对比页即使请求成功率为 0，也不容易从 UI 侧第一时间看出是“参数语义错了”。

### 3.3 菜谱详情 DTO 存在字段预期差异

- 前端 `RecipeDetailDTO` 里保留了 `story`、顶层 `ingredients`、顶层 `steps` 等字段，见 [recipe.ts](/Users/xychen/Documents/MyCode/menu-time/miniapp/src/services/types/recipe.ts)。
- 后端 `RecipeDetailDto` 的核心内容在 `currentVersion` 下，且提供 `slug / coverSource / status / latestMomentAt` 等字段，见 [recipes.types.ts](/Users/xychen/Documents/MyCode/menu-time/backend/src/server/modules/recipes/recipes.types.ts)。

影响：

- 详情页、编辑页、版本页可能出现映射层补丁越来越多的问题。

### 3.4 Taxonomy 真实能力大于前端当前接入能力

- 后端已支持 `DELETE` 和 `reorder`
- 前端当前 service 仅接了 `get/create/update`

影响：

- 不阻塞本轮主联调，但要在计划里明确这是“后续补齐项”，避免误判为接口不可用。

## 4. 推荐联调顺序

### 4.1 为什么这样排

当前最短、最稳的业务闭环是：

`登录 -> 拉取分类/标签 -> 菜谱库 -> 新建菜谱 -> 进入详情 -> 新建版本 -> 版本对比 -> 设为当前版本`

原因：

- 这条链路前后端都已经具备能力。
- 它覆盖了阶段 2 和阶段 3 最核心的真实用户路径。
- 它能尽早暴露 DTO、分页、鉴权、缓存失效、页面跳转四类高风险问题。
- 它不会被 moments / meal-plans 等未完成域拖住。

## 5. 联调任务计划清单

### 任务 0：接口契约冻结与差异收口

目标：

- 在正式切真接口前，先把当前最明显的前后端契约差异收口。

任务项：

- 对齐 recipes 相关 DTO，给出“后端响应 -> 前端 ViewModel”的最终映射规则。
- 决定 `/api/v1/recipes/:id/compare` 继续使用“版本号”还是改成“版本 ID”。
- 对齐 `/api/v1/recipes/:id/versions` 的分页结构消费方式。
- 明确 auth、taxonomy、recipes 的错误码与错误提示映射。
- 以 Swagger 为准，补一版前端联调对照表。

建议产出：

- 一份接口差异清单
- 一份前端映射规则表
- 必要的前后端小改动 PR

完成标准：

- 前端服务层不再依赖猜测字段。
- 版本列表、版本对比、详情页三处的字段语义明确。

### 任务 1：联调环境与账号链路打通

目标：

- 让前端开发、后端开发、测试三方在同一环境上重复走通登录。

任务项：

- 准备可访问数据库与完整 `backend/.env` / `backend/.env.local`
- 确认后端本地可执行 migration、seed、dev server
- 确认小程序 `apiBaseUrl` 指向当前联调环境
- 使用 `mock:<openid>` 跑通微信登录模拟链路
- 验证 access token 注入、401 自动 refresh、refresh 失败登出

完成标准：

- 真机或开发者工具首次进入能拿到 session
- Token 过期后可自动刷新
- 刷新失败后会清理本地状态并回到匿名态

### 任务 2：Auth + Taxonomy 联调并关停对应 mock

目标：

- 先把最基础的用户态与字典数据接稳，为菜谱域提供真实筛选基础。

任务项：

- 联调 `POST /api/v1/auth/wechat-login`
- 联调 `POST /api/v1/auth/refresh`
- 联调 `GET /api/v1/auth/session`
- 联调 `GET /api/v1/categories`
- 联调 `GET /api/v1/tags`
- 联调分类管理页的新建/更新
- 联调标签管理页的新建/更新
- 确认筛选组件、表单选择器、缓存刷新表现

完成标准：

- 菜谱库筛选项来自真接口，不再来自 taxonomy mock
- 分类/标签管理页保存后，列表和相关页面筛选项同步刷新

### 任务 3：菜谱库列表与详情真接口联调

目标：

- 让用户能在真实数据下浏览和查看菜谱。

任务项：

- 联调 `GET /api/v1/recipes`
- 联调 `GET /api/v1/recipes/:id`
- 对齐分页、搜索、分类筛选、标签筛选入参
- 对齐详情页顶部信息区字段映射
- 对齐 `currentVersion` 的分类、标签、食材、步骤展示
- 校验空列表、无搜索结果、接口报错场景

完成标准：

- 菜谱库可在真实数据库数据下正常分页和筛选
- 点击菜谱卡片进入详情页后，信息展示正确且不依赖 mock

### 任务 4：菜谱创建与基础信息编辑联调

目标：

- 打通“创建菜谱 -> 回跳详情 -> 基础信息编辑”的真实闭环。

任务项：

- 联调 `POST /api/v1/recipes`
- 联调 `PATCH /api/v1/recipes/:id`
- 校验“选择现有分类/标签 + 临时新建”交互
- 校验表单字段与后端 schema 的一一对应关系
- 对齐成功后缓存失效和回跳策略
- 对齐 400/401/409 错误提示文案

完成标准：

- 能新建一条真实菜谱并进入详情页
- 能更新名称等基础信息并刷新成功
- 创建/更新后菜谱库与详情页状态一致

### 任务 5：版本列表、版本创建、版本对比、设为当前版本联调

目标：

- 完成阶段 2 与后端阶段 3 最核心的一段真接口闭环。

任务项：

- 联调 `GET /api/v1/recipes/:id/versions`
- 联调 `POST /api/v1/recipes/:id/versions`
- 联调 `GET /api/v1/recipes/:id/versions/:versionId`
- 联调 `GET /api/v1/recipes/:id/compare`
- 联调 `POST /api/v1/recipes/:id/versions/:versionId/set-current`
- 修复版本列表分页消费问题
- 修复版本对比参数语义问题
- 校验“发布新版本 -> 回到详情 -> 版本列表刷新 -> 进入对比 -> 设为当前版本”的链路

完成标准：

- 新建版本后，版本列表出现新增项
- 对比页摘要、版本内容和设为当前版本动作都正确
- 切换当前版本后，详情页展示的新版本内容正确

### 任务 6：Recipes 域去 mock 与回归验证

目标：

- 关闭 recipes 相关 mock，避免真接口接入后仍被假数据掩盖问题。

任务项：

- 下线 recipes 相关 mock 开关或将其限制为显式 debug 模式
- 回归菜谱库、详情、编辑、版本创建、版本对比全链路
- 验证 query key、缓存失效、返回跳转、页面重入
- 记录一份联调缺陷清单与修复结果

完成标准：

- recipes 域主链路默认走真接口
- 同一页面不存在“部分真接口 + 部分旧 mock”混跑

### 任务 7：阶段性验收与下阶段交接

目标：

- 在当前可联调范围内形成阶段验收结论，并给下一阶段输入清晰边界。

任务项：

- 完成一轮页面走查
- 完成一轮开发者工具联调
- 完成一轮真机关键路径验证
- 输出阶段验收记录
- 明确下一阶段开始条件：moments/media 接口 ready 后再进入阶段 3 联调

完成标准：

- 当前阶段形成“已完成联调 / 遗留问题 / 下一阶段前置条件”三段式结论

## 6. 任务优先级

### P0：必须先做

- 任务 0：接口契约冻结与差异收口
- 任务 1：联调环境与账号链路打通
- 任务 2：Auth + Taxonomy 联调

### P1：本轮核心价值

- 任务 3：菜谱库列表与详情联调
- 任务 4：菜谱创建与基础信息编辑联调
- 任务 5：版本主链路联调

### P2：本轮收尾

- 任务 6：Recipes 域去 mock 与回归验证
- 任务 7：阶段性验收与下阶段交接

## 8. 本轮不纳入联调范围的事项

- 时光记录、图片上传、首页动态流
- 周菜单、点菜台增删改查
- 购物清单聚合与勾选
- 随机点菜
- 家庭协作与权限差异

这些能力当前不是“最合适联调任务”，继续纳入只会把团队注意力从已完成域上拉走。

## 9. 结论

在当前阶段下，最合适的联调计划不是继续铺新能力，而是优先把已完成的 `Auth + Taxonomy + Recipes + Versions` 四个能力域彻底接稳。

一句话版本：

先收口契约，再打通登录和字典，再跑通“菜谱创建到版本切换”的最短真实闭环，最后去 mock 和做回归，这就是当前最划算、风险最低、收益最大的联调顺序。
