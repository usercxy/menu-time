# 食光记前后端接口契约差异清单

## 1. 目的

本文档用于支撑 [前后端联调任务计划清单.md](/Users/xychen/Documents/MyCode/menu-time/docs/前后端联调任务计划清单.md) 中的“任务 0：接口契约冻结与差异收口”。

目标不是罗列所有接口，而是优先标出会直接阻塞当前联调主链路的问题，并给出建议收口方案。

## 2. 结论先看

当前最需要优先收口的契约差异一共有 4 类：

1. Auth 返回结构不一致
2. Recipe Detail 结构不一致
3. Recipe Versions 列表分页结构不一致
4. Recipe Compare 参数语义不一致

这 4 类如果不先处理，前端一切真接口，登录、详情页、版本页、对比页都会出现直接报错或隐性错渲染。

## 3. 差异清单

### 3.1 Auth：登录与刷新返回结构不一致

#### 前端当前定义

- [auth.ts](/Users/xychen/Documents/MyCode/menu-time/miniapp/src/services/types/auth.ts)
- [bootstrap.tsx](/Users/xychen/Documents/MyCode/menu-time/miniapp/src/features/auth/bootstrap.tsx)

前端当前假设：

- `POST /api/v1/auth/wechat-login` 返回扁平结构：
  - `accessToken`
  - `refreshToken`
  - `expiresIn`
  - `user`
- `POST /api/v1/auth/refresh` 返回：
  - `accessToken`
  - `refreshToken`
  - `expiresIn`
- `GET /api/v1/auth/session` 返回：
  - `session.user`

#### 后端当前定义

- [auth.types.ts](/Users/xychen/Documents/MyCode/menu-time/backend/src/server/modules/auth/auth.types.ts)
- [auth.mapper.ts](/Users/xychen/Documents/MyCode/menu-time/backend/src/server/modules/auth/auth.mapper.ts)

后端实际返回：

- `POST /api/v1/auth/wechat-login`
  - `session`
  - `tokens.accessToken`
  - `tokens.refreshToken`
  - `tokens.accessTokenExpiresIn`
  - `tokens.refreshTokenExpiresIn`
- `POST /api/v1/auth/refresh`
  - 同上
- `GET /api/v1/auth/session`
  - `userId`
  - `householdId`
  - `householdName`
  - `nickname`
  - `role`

#### 风险

- 前端登录启动逻辑在 [bootstrap.tsx](/Users/xychen/Documents/MyCode/menu-time/miniapp/src/features/auth/bootstrap.tsx) 中直接读取 `loginPayload.accessToken`、`loginPayload.user`，接真接口后会立即取不到字段。
- `refreshAccessToken()` 在 [client.ts](/Users/xychen/Documents/MyCode/menu-time/miniapp/src/services/request/client.ts) 中把 refresh 结果直接当作 `TokenBundleDTO` 写回本地缓存，接真接口后同样会失败。
- `SessionDTO` 当前是 `session.user` 包一层，和后端真实返回不一致。

#### 建议收口方案

推荐以前端适配后端为主，原因是后端当前结构更完整，也更清晰地区分了 `session` 和 `tokens`。

建议修改前端：

- `WechatLoginDTO` 改为：
  - `session`
  - `tokens`
- `TokenBundleDTO` 改为：
  - `accessToken`
  - `refreshToken`
  - `accessTokenExpiresIn`
  - `refreshTokenExpiresIn`
- `SessionDTO` 改为直接对应后端：
  - `userId`
  - `householdId`
  - `householdName`
  - `nickname`
  - `role`
- `bootstrap.tsx`、`client.ts`、`store/session.ts` 同步改消费方式

优先级：

- `P0`

### 3.2 Recipe Detail：详情 DTO 结构不一致

#### 前端当前定义

- [recipe.ts](/Users/xychen/Documents/MyCode/menu-time/miniapp/src/services/types/recipe.ts)

前端当前 `RecipeDetailDTO` 假设包含：

- `story`
- 顶层 `ingredients`
- 顶层 `steps`
- 顶层 `tips`
- `currentVersion`

#### 后端当前定义

- [recipes.types.ts](/Users/xychen/Documents/MyCode/menu-time/backend/src/server/modules/recipes/recipes.types.ts)

后端当前 `RecipeDetailDto` 实际包含：

- `id`
- `name`
- `slug`
- `coverImageUrl`
- `coverSource`
- `versionCount`
- `momentCount`
- `latestMomentAt`
- `latestCookedAt`
- `status`
- `currentVersion`

而版本内容字段：

- `ingredients`
- `steps`
- `tips`
- `category`
- `tags`

都在 `currentVersion` 内部。

#### 风险

- 详情页、编辑页如果还按顶层 `ingredients/steps/tips` 取值，会出现空渲染或兼容补丁越来越多。
- `story` 在后端当前并不存在，前端若继续视为必需字段会导致页面模型失真。

#### 建议收口方案

推荐前端收口为“两层模型”：

- DTO 层完全贴合后端
- ViewModel 层再为页面整理出：
  - `displayIngredients = currentVersion?.ingredients ?? []`
  - `displaySteps = currentVersion?.steps ?? []`
  - `displayTips = currentVersion?.tips ?? ''`

同时：

- 从前端类型中移除或改为可选 `story`
- 不再要求顶层 `ingredients / steps / tips`

优先级：

- `P0`

### 3.3 Recipe Versions：版本列表分页结构不一致

#### 前端当前定义

- [recipe.ts](/Users/xychen/Documents/MyCode/menu-time/miniapp/src/services/modules/recipe.ts)

前端当前：

- `getRecipeVersions(recipeId)` 直接返回 `RecipeVersionListItemDTO[]`

#### 后端当前定义

- [recipes.types.ts](/Users/xychen/Documents/MyCode/menu-time/backend/src/server/modules/recipes/recipes.types.ts)

后端当前：

- `GET /api/v1/recipes/:id/versions` 返回 `PageResult<RecipeVersionListItemDto>`

并且后端接口还支持：

- `page`
- `pageSize`

#### 风险

- 版本列表页接真接口后，前端会把分页对象当数组用。
- 后续如果版本数增长，前端没有分页意识，页面行为会不稳定。

#### 建议收口方案

推荐前端对齐后端分页，不建议让后端为了前端临时退回数组结构。

建议修改前端：

- `getRecipeVersions(recipeId, query?)`
- 返回 `PageResult<RecipeVersionListItemDTO>`
- 页面层先消费 `result.items`
- 如果当前 UI 不准备做分页，也至少保留 `page/pageSize/hasMore`，避免再次返工

优先级：

- `P0`

### 3.4 Recipe Compare：参数语义不一致

#### 前端当前定义

- [recipe.ts](/Users/xychen/Documents/MyCode/menu-time/miniapp/src/services/modules/recipe.ts)

前端当前：

- `compareRecipeVersions(recipeId, baseVersionId, targetVersionId)`
- 请求参数是：
  - `base: baseVersionId`
  - `target: targetVersionId`

#### 后端当前定义

- [recipes.schema.ts](/Users/xychen/Documents/MyCode/menu-time/backend/src/server/modules/recipes/recipes.schema.ts)

后端当前：

- `GET /api/v1/recipes/:id/compare`
- 参数校验要求：
  - `base` 为整数版本号
  - `target` 为整数版本号

#### 风险

- 前端传 UUID，后端要 number，接口会直接 400。
- 这个问题从 UI 表面看像“对比页坏了”，但根因其实是契约不一致。

#### 建议收口方案

推荐优先改前端，因为：

- 后端 compare 逻辑天然更适合用版本号表达“V1 vs V2”
- 当前后端 DTO 里版本详情本来就包含 `versionNumber`

前端建议改为：

- `compareRecipeVersions(recipeId, baseVersionNumber, targetVersionNumber)`
- 参数类型改成 `number`
- 版本列表点击“对比”时，从选中项中传 `versionNumber`

如果产品层明确要求“只认 versionId”，那就再统一改后端，但不建议在联调中间保留双语义。

优先级：

- `P0`

### 3.5 Taxonomy：写接口字段范围不完全一致

#### 前端当前定义

- [taxonomy.ts](/Users/xychen/Documents/MyCode/menu-time/miniapp/src/services/types/taxonomy.ts)

前端当前写入字段：

- Category：
  - `name`
  - `color`
- Tag：
  - `name`

#### 后端当前定义

- [taxonomy.types.ts](/Users/xychen/Documents/MyCode/menu-time/backend/src/server/modules/taxonomy/taxonomy.types.ts)

后端还支持：

- Category：
  - `sortOrder`
- Tag：
  - `sortOrder`
- 以及 `DELETE`、`reorder`

#### 风险

- 不阻塞本轮主联调
- 但如果前端误以为后端“不支持排序/删除”，会影响后续阶段安排

#### 建议收口方案

本轮不强制改动。

本轮仅需：

- 在联调文档里明确 taxonomy 真接口已支持更多能力
- 前端先完成当前已上线页面所需的 `get/create/update`
- 后续做管理页增强时再补 `delete/reorder`

优先级：

- `P2`

### 3.6 Recipes Filter：标签筛选命名不一致

#### 前端当前定义

- [recipe.ts](/Users/xychen/Documents/MyCode/menu-time/miniapp/src/services/types/recipe.ts)

前端当前查询字段：

- `tagId?: string`

#### 后端当前定义

- [recipes.schema.ts](/Users/xychen/Documents/MyCode/menu-time/backend/src/server/modules/recipes/recipes.schema.ts)

后端当前查询字段：

- `tagIds`
- 格式为逗号分隔字符串，支持多标签

#### 风险

- 当前如果前端只传单个 `tagId`，后端不会按预期过滤。
- 会造成“筛选 UI 看起来能选，列表却不变化”的假性故障。

#### 建议收口方案

前端改为：

- 查询类型字段使用 `tagIds?: string[]`
- service 层统一序列化为逗号分隔字符串
- 页面层即使暂时只支持单选，也通过数组传递，避免未来再改接口

优先级：

- `P1`

## 4. 推荐收口顺序

### 第一步：先收 P0

- Auth 返回结构
- Recipe Detail 结构
- Recipe Versions 分页结构
- Recipe Compare 参数语义

### 第二步：再收 P1

- Recipes 标签筛选命名

### 第三步：记录 P2，暂不阻塞

- Taxonomy delete / reorder / sortOrder 的前端接入

## 5. 推荐分工

### 前端

- 修改 `services/types`
- 修改 `services/modules`
- 修改 auth bootstrap、request refresh、recipe detail / version compare 页面消费方式
- 移除与真实契约冲突的 mock 假设

### 后端

- 用 Swagger 固化当前最终契约
- 如有必要，补充响应体字段说明
- 确认 compare 接口最终采用“版本号”还是“版本 ID”

### 联调共同任务

- 按真实接口各跑一遍：
  - 登录
  - 拉分类/标签
  - 菜谱列表
  - 菜谱详情
  - 创建菜谱
  - 新建版本
  - 对比版本
  - 设为当前版本

## 6. 最终建议

当前最值得做的不是继续加新接口，而是先把前端服务层和后端真实契约彻底贴齐。

如果只做一件事，建议先改 `auth` 契约；如果做两件事，再把 `versions` 分页和 `compare` 参数一起收掉。这样切真接口时，主链路就不会在最前面几步就断掉。
