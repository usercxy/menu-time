# 食光记后端开发规范

## 1. 文档说明

本文档用于统一后端开发过程中的工程结构、编码风格、接口实现、数据库访问、测试和发布规范，减少以下问题：

- 路由层、服务层、数据库层职责混乱
- 事务、错误码、日志、鉴权风格不一致
- 联调时接口结构反复变化
- 软删除、家庭隔离、快照规则被遗漏

适用技术栈：

- `Next.js App Router`
- `TypeScript`
- `PostgreSQL`
- `Prisma`
- `Zod`
- `Auth.js`
- `Pino`
- `pg-boss`

## 2. 核心原则

- 单体优先：MVP 阶段以单体后端完成闭环，不引入微服务复杂度。
- 数据先稳：先保证数据模型、索引、事务边界正确，再追求花哨实现。
- 快照优先：版本、菜单、购物清单都要能回放历史。
- 规则后置：核心业务规则必须放在服务层，不交给前端拼。
- 隔离优先：所有业务数据默认带 `household_id`。

## 3. 工程结构规范

### 3.1 推荐目录

```text
src/
  app/
    api/
      health/
      v1/
        auth/
        categories/
        tags/
        recipes/
        moments/
        menu-plans/
        random-picks/
        shopping-lists/
        media/
  server/
    db/
      client.ts
      transactions.ts
    lib/
      auth/
      env/
      errors/
      jobs/
      logger/
      storage/
      zod/
    modules/
      auth/
      recipes/
      moments/
      plans/
      random/
      shopping/
      taxonomy/
  types/
prisma/
  schema.prisma
  migrations/
tests/
```

### 3.2 分层职责

Route Handler 负责：

- 接收请求
- 获取会话
- Zod 校验
- 调用 service
- 返回统一响应

Service 负责：

- 业务规则
- 事务边界
- 状态流转
- 冗余字段回写
- 触发异步任务

Repository / Prisma 负责：

- 数据访问
- 查询封装
- 批量写入

禁止行为：

- 在 Route Handler 里直接写复杂 Prisma 逻辑
- 在前端拼版本号、购物清单聚合、随机点菜规则
- 在 mapper 中做数据库写入

## 4. 命名与文件规范

- 文件名使用小写加模块后缀，如 `recipe.service.ts`
- TypeScript 类型、类、枚举使用 PascalCase
- 变量、函数使用 camelCase
- 数据库字段使用 snake_case
- API 响应字段使用 camelCase

建议每个模块包含：

- `xxx.service.ts`
- `xxx.repository.ts`
- `xxx.schema.ts`
- `xxx.mapper.ts`
- `xxx.types.ts`

## 5. API 开发规范

### 5.1 路径与方法

- API 前缀统一 `/api/v1`
- 资源名统一使用复数，如 `/recipes`
- 获取资源用 `GET`
- 创建资源用 `POST`
- 局部更新用 `PATCH`
- 删除资源用 `DELETE`

动作型接口仅在必要时使用，如：

- `/recipes/:id/versions/:versionId/set-current`
- `/random-picks/sessions/:id/results/:resultId/accept`

### 5.2 响应结构

所有接口必须返回统一壳：

```json
{
  "success": true,
  "data": {},
  "requestId": "req_xxx"
}
```

错误统一返回：

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "字段错误",
    "details": null
  },
  "requestId": "req_xxx"
}
```

### 5.3 DTO 规范

- 输入 DTO 全部由 Zod 定义并校验
- 输出 DTO 通过 mapper 显式组装
- 不允许直接把 Prisma Model 原样返回给前端

## 6. 参数校验规范

- 每个模块的请求结构定义在 `xxx.schema.ts`
- 路由层只做 `safeParse` 和错误转换
- 复杂跨字段校验使用 `superRefine`

必须校验：

- UUID 格式
- 必填字段
- 长度限制
- 日期格式
- 评分范围
- 数组长度上限
- 枚举值合法性

特别校验：

- 时光图片最多 9 张
- `weekStartDate` 必须是周一
- 菜单项日期必须落在目标周内
- `recipeVersionId` 必须属于对应 `recipeId`

## 7. 错误处理规范

推荐错误码：

- `VALIDATION_ERROR`
- `UNAUTHORIZED`
- `FORBIDDEN`
- `NOT_FOUND`
- `CONFLICT`
- `BUSINESS_RULE_VIOLATION`
- `INTERNAL_ERROR`

建议封装统一异常，例如：

```ts
throw new AppError('NOT_FOUND', '菜谱不存在', 404)
```

要求：

- 业务层抛业务异常
- 路由层或全局处理器负责转为 HTTP 响应
- 不向前端暴露 Prisma 原始错误和堆栈

## 8. 鉴权与权限规范

- 除 `auth/login` 和 `health` 外，所有接口默认需要登录
- 会话中至少包含 `userId`、`householdId`、`role`
- `householdId` 只能来自服务端会话，不接受前端传入
- 所有查询和写入必须带 `householdId` 约束

虽然 MVP 只有管理员，代码仍需预留角色判断入口：

- `admin`：完整写权限
- `member`：二期按接口逐步放开

## 9. 数据库与 Prisma 规范

### 9.1 查询规范

- 列表接口优先查摘要字段，不返回大文本明细
- 默认显式选择字段，避免过度 `include`
- 软删除实体默认过滤 `deleted_at IS NULL`
- 索引字段优先用于排序和分页

### 9.2 写入规范

- 多表写入必须使用事务
- 冗余字段回写与主写入放在同一事务中
- 批量步骤、食材、标签关系优先批量写入

### 9.3 Migration 规范

- 每次表结构变更必须配套 migration
- 不直接手改数据库结构
- 常规索引写在 Prisma Schema
- 部分唯一索引、check 约束等用 SQL migration 补充

## 10. 事务规范

必须使用事务的场景：

- 创建菜谱并生成 V1
- 新建版本并切换当前版本
- 新增时光记录并更新菜谱统计
- 生成购物清单快照
- 接受随机点菜结果并写入周菜单

事务边界原则：

- 事务内只做强一致写入
- 图片处理、分享图生成、通知等放事务外或异步任务
- 不在事务内做网络请求

并发处理要求：

- 版本号递增要有唯一索引兜底
- 菜单排序冲突要按范围重排
- 购物清单版本号冲突要可重试

## 11. 日志与可观测性规范

统一使用结构化日志，推荐 Pino。

关键日志字段：

- `requestId`
- `userId`
- `householdId`
- `module`
- `action`
- `durationMs`

日志级别：

- `info`：正常业务节点
- `warn`：可恢复异常、冲突、重试
- `error`：接口失败、未处理异常、任务失败

禁止记录：

- 明文密码
- 完整 Cookie
- 敏感 Token
- 大体积二进制内容

监控要求：

- 提供 `/api/health`
- 接入 Sentry 记录运行时异常
- 异步任务要记录成功率与失败原因

## 12. 文件上传与异步任务规范

上传统一走两段式：

1. 请求上传凭证
2. 客户端直传对象存储
3. 调用完成接口登记 `media_assets`

适合异步化的任务：

- 菜谱封面自动回填
- 分享图生成
- 图片衍生图处理
- 统计修复

Job 设计要求：

- 参数尽量只传 ID
- 任务实现必须幂等
- 失败支持重试
- 关键状态变化要有日志

## 13. 测试规范

单元测试优先覆盖：

- 版本差异摘要生成
- 食材聚合与调料分类
- 随机点菜候选过滤
- 菜单日期校验

集成测试至少覆盖：

- 登录与鉴权
- 创建菜谱 -> 新建版本 -> 切换当前版本
- 新增时光记录 -> 更新菜谱统计
- 周菜单 -> 生成购物清单
- 随机点菜接受结果 -> 写入菜单

联调前确认：

- 响应格式统一
- 错误码稳定
- 日期格式统一
- 软删除资源默认不返回

## 14. 环境变量与配置规范

必备环境变量：

- `DATABASE_URL`
- `AUTH_SECRET`
- `APP_BASE_URL`
- `S3_ENDPOINT`
- `S3_BUCKET`
- `S3_ACCESS_KEY`
- `S3_SECRET_KEY`
- `SENTRY_DSN`

要求：

- 统一通过 `env` 模块读取和校验
- 启动时一次性校验，不在业务代码中零散读取 `process.env`
- 本地、测试、生产的差异要文档化

## 15. 代码评审清单

提交前至少自查：

- 是否遵守统一响应结构
- 是否做了 Zod 校验
- 是否带了 `householdId` 过滤
- 是否遗漏软删除过滤
- 是否把多表写入包进事务
- 是否记录关键日志
- 是否直接暴露 Prisma 原始模型
- 是否补充必要测试

## 16. MVP 阶段必须坚持的底线

- 菜谱版本必须独立建模
- 周菜单必须绑定具体版本
- 购物清单必须快照化
- 所有主数据必须保留 `household_id`
- 核心业务规则必须放在服务层
