# 食光记后端阶段 3 开发计划清单

## 1. 文档目标

本文档用于将“完成后端阶段 3：菜谱与版本管理”为止的推荐开发顺序细化为可执行任务清单，方便按里程碑推进、逐项验收。

适用范围：

- 以 `backend/` 当前现状为起点推进开发。
- 覆盖“阶段 3 完成”所需的最小闭环。
- 强调每个里程碑完成后都要进行测试，确认服务可正常运行。

本文档不替代以下设计文档，仅负责把工作进一步拆细：

- `docs/backend/后端开发TODO.md`
- `docs/backend/后端技术方案.md`
- `docs/backend/数据库详细设计.md`
- `docs/backend/API 接口清单.md`

## 2. 当前基线

基于当前代码与文档状态，后端已具备如下基础能力：

- 已完成工程底座、统一路由包装器、统一响应、统一错误处理、日志、环境变量校验、Prisma Client、事务封装。
- 已完成微信登录、刷新、退出、当前会话查询 4 个鉴权接口。
- 已完成 `/api/health` 健康检查接口。
- 已完成 taxonomy 模块与分类/标签 API，可作为菜谱录入依赖数据源。
- 已完成 recipes 域 Prisma 模型、migration，以及 `recipes.types / schema / mapper / repository / service` 模块骨架。
- 已在 `src/server/lib/diff` 下补齐版本 diff helper，可覆盖主料变化、标签新增/移除、步骤数变化。
- 已完成 recipes service 层主链路：列表、详情、创建菜谱自动生成 `V1`、更新基础信息、软删除。
- 已完成 recipes service 层版本主链路：版本列表、版本详情、创建版本、版本对比、切换当前版本。

当前阶段 3 的关键现实情况：

- 已完成 `recipes` 相关 Route Handler 接入，前端可以通过 HTTP 调用菜谱与版本主链路。
- 已补齐本地联调所需 recipes seed 数据与示例请求。
- 已完成阶段 3 自测与验收，阶段 4 可在此基础上继续推进。

因此，阶段 3 已形成“数据模型 -> 模块抽象 -> service 主链路 -> API 路由 -> seed/验收”的最小闭环。

## 3. 总体开发顺序

建议按以下顺序推进：

1. 里程碑 A：补齐阶段 3 依赖的数据模型与 migration。
2. 里程碑 B：建立 recipes 模块内部基础抽象。
3. 里程碑 C：实现菜谱主表 CRUD 与当前版本详情链路。
4. 里程碑 D：实现版本管理、版本对比与当前版本切换。
5. 里程碑 E：接入 recipes API 路由。
6. 里程碑 F：补 seed、自测、联调样例与阶段 3 验收。

原则：

- 先定表结构与事务边界，再写业务接口。
- 先打通“创建菜谱自动生成 V1”，再扩展版本能力。
- 所有 recipes 域查询和写入必须显式带 `householdId`。
- 菜谱当前展示内容统一以 `recipes.current_version_id` 指向版本为准。
- 每个里程碑完成后必须做一次测试，未通过则不进入下个里程碑。
