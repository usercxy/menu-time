# 食光记后端阶段 2 开发计划清单

## 1. 文档目标

本文档用于将“完成后端阶段 2”为止的推荐开发顺序细化为可执行任务清单，方便按里程碑推进、逐项验收。

适用范围：

- 以 `backend/` 当前现状为起点推进开发。
- 覆盖“阶段 2 完成”所需的最小闭环。
- 强调每个里程碑完成后都要进行测试，确认服务可正常运行。

本文档不替代以下设计文档，仅负责把工作进一步拆细：

- `docs/backend/后端开发TODO.md`
- `docs/backend/后端技术方案.md`
- `docs/backend/数据库详细设计.md`
- `docs/backend/API 接口清单.md`

## 2. 当前基线

基于当前代码状态，后端已具备如下基础能力：

- 已完成工程底座、统一路由包装器、统一响应、统一错误处理、日志、环境变量校验、Prisma Client、事务封装。
- 已完成微信登录、刷新、退出、当前会话查询 4 个鉴权接口。
- 已完成 `/api/health` 健康检查接口。
- 已完成 taxonomy 模块的 `schema / mapper / service / repository` 与 API 路由。
- Prisma 已落地 `households`、`users`、`wechat_accounts`、`refresh_tokens`、`categories`、`tags`，并包含首个 migration。

当前要达成的目标不是继续扩展阶段 3，而是先补齐阶段 2 的两个核心缺口：

1. 查询层统一 `household_id` 隔离。
2. 分类 / 标签主数据接口完整可用。

## 3. 总体开发顺序

建议按以下顺序推进：

1. 里程碑 A：补齐阶段 2 所需的最小数据模型与 migration。
2. 里程碑 B：建立 household 查询隔离基础设施。
3. 里程碑 C：实现 taxonomy 模块本体。
4. 里程碑 D：接入 taxonomy API 路由。
5. 里程碑 E：补 seed、完成联调前自测与阶段 2 验收。

原则：

- 先定数据结构，再写业务接口。
- 先定查询隔离约定，再铺具体 CRUD。
- 先做分类，再复用同一模式做标签。
- 每个里程碑完成后必须做一次测试，未通过则不进入下个里程碑。

## 7. 推荐执行节奏

建议按以下节奏推进，每完成一步就做验证：

1. 完成里程碑 A，先验证数据库层。
2. 完成里程碑 B，先验证隔离策略。
3. 完成里程碑 C，先验证 service 层逻辑。
4. 完成里程碑 D，验证 API 行为和返回格式。
5. 完成里程碑 E，做整体验收后再进入阶段 3。

如果某个里程碑测试未通过，不建议继续下一阶段开发，应先修复并重新验证。
