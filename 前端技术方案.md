## 2. 产品范围与前端落地策略

### 2.1 业务目标

前端首要目标不是“完整做完所有页面”，而是优先打通以下闭环：

1. 创建菜谱
2. 生成 V1 版本
3. 进入菜谱详情查看当前做法
4. 补充时光记录
5. 加入本周菜单
6. 生成购物清单

### 2.2 页面原型与需求对齐结论

结合 PRD 和 `stitch_prd`，前端页面分为三层：

- 核心闭环页：首页、菜谱库、菜谱详情、菜谱新建/编辑、新建版本、记一笔、点菜台、购物清单、我的
- 管理配置页：分类管理、标签体系
- 增强能力页：版本对比、随机点菜、全局时光轴、家庭协作

### 2.3 版本策略建议

考虑到仓库内 [TODO.md](D:/AI/Menu Time/TODO.md) 的开发节奏与原型完整度，建议前端按两阶段落地：

- 阶段 A：MVP 主闭环
  - 首页
  - 菜谱库
  - 菜谱新建/编辑
  - 菜谱详情
  - 新建版本
  - 菜谱内时光轴
  - 记一笔
  - 点菜台
  - 购物清单
  - 我的
  - 分类管理
  - 标签管理
- 阶段 B：增强体验
  - 版本对比独立页
  - 随机点菜页
  - 全局时光轴
  - 家庭协作

说明：PRD 中“随机点菜”为 P0，原型和接口也已具备，因此前端工程与数据模型需要一次性预留该模块，但实现优先级可略后于核心闭环。

## 3. 技术选型

### 3.1 推荐栈

| 层级 | 方案 | 说明 |
| --- | --- | --- |
| 框架 | Taro + React + TypeScript | 兼顾小程序原生能力与工程化组织 |
| 路由 | Taro 页面路由 + tabBar + 分包 | 符合小程序形态 |
| 服务端状态 | TanStack Query | 统一列表缓存、详情缓存、失效刷新 |
| 客户端状态 | Zustand | 处理会话、筛选草稿、页面临时共享状态 |
| 校验 | Zod | 表单校验与接口返回兜底解析 |
| 样式 | SCSS Modules + Design Tokens | 不直接把原型中的 Tailwind 类带入生产代码 |
| 请求 | `Taro.request` 二次封装 | 统一鉴权、错误处理、重试、日志 |
| 上传 | `Taro.uploadFile` + 媒体登记接口 | 对齐后端媒体上传流程 |
| 时间处理 | dayjs | 周历、日期格式化、相对时间 |

### 3.2 组件库选型结论

结合当前技术路线 `Taro + React + 微信小程序`，以及各组件库官方文档的兼容说明，建议采用“主库 + 补充库 + 自定义业务组件”的组合，而不是完全依赖单一组件库。

推荐结论如下：

| 角色 | 方案 | 推荐度 | 结论 |
| --- | --- | --- | --- |
| 主组件库 | `Taroify` | 高 | 最适合作为本项目主组件库 |
| 备选组件库 | `@antmjs/vantui` | 中 | 适合作为 Vant 体系兼容备选 |
| 原生小程序组件库 | `Vant Weapp` | 中低 | 仅适合局部、强原生场景，不建议做主方案 |
| Vue 向方案 | `NutUI` | 低 | 当前官方接入说明更偏 `Taro + Vue3`，不适合本项目当前 React 路线 |
| 设计参考库 | `TDesign 微信小程序` | 中 | 可借鉴其交互模式与页面组织，但不建议直接作为 Taro React 主库 |

### 3.3 为什么主选 Taroify

`Taroify` 官方文档说明它是移动端组件库 Vant 的 Taro React 版本，支持 React、按需引入、Tree Shaking、主题定制，并提供 70+ 组件与 700+ 主题变量，整体与本项目的技术栈和设计可塑性最匹配。

它尤其适合承接以下通用能力：

- `Tabs`、`Tabbar`、`Navbar`
- `Popup`、`ActionSheet`、`Picker`
- `List`、`Empty`、`Skeleton`
- `Tag`、`Badge`、`Avatar`
- `Sticky`、`Swiper`
- `Timeline`
- `FloatingBubble`

对“食光记”这种偏内容卡片、弹层操作、表单编辑、时间轴展示较多的小程序，Taroify 能覆盖大部分基础交互组件，而不强行把视觉风格锁死在标准电商模板上。

### 3.4 为什么不把 NutUI 作为主方案

Taro 官方当前文档中对 NutUI 的接入说明，重点落在 `Taro 3.5+ + Vue3` 方案，并以 `@nutui/nutui-taro` 和 Vue 组件自动注册为主。这意味着：

- 它和本文档既定的 `Taro + React` 路线不一致
- 如果中途切 NutUI，等于要连组件体系和开发范式一起调整
- 对当前项目来说，迁移成本高于收益

因此不建议把 NutUI 写入主技术路线。

### 3.5 为什么不把 Vant Weapp 作为主方案

Taro 官方在 `Vant Weapp` 接入文档中明确提示：使用原生第三方组件库进行开发的项目，将不再具有多端转换能力。

这对本项目的影响是：

- 如果后续需要保留 H5 / 其他小程序端扩展空间，会被锁死
- 工程配置会更复杂，需要 `usingComponents`、原生资源 copy、尺寸转换黑名单等额外处理
- 页面中 React 组件与原生小程序组件混搭的调试成本更高

因此 `Vant Weapp` 只建议作为局部补充方案，而不是全局主库。

适合局部引入的场景：

- 某些成熟度极高、原生体验更稳的表单或选择类组件
- 某些 Taro React 生态暂时缺少稳定替代的原生小程序能力封装

### 3.6 备选方案：@antmjs/vantui

Taro 官方在 Vant Weapp 文档中推荐过 `@antmjs/vantui`，它的定位是基于 Vant Weapp 的多端组件库，同时支持 Taro 和 React。

它的优势在于：

- React 友好
- 与 Vant 视觉和 API 体系高度一致
- 比直接接入 `Vant Weapp` 更适合 Taro 工程

但从本项目角度看，它更适合作为备选和兜底，而不是第一选择，原因是：

- 社区体量和资料丰富度略弱于 Vant 主生态
- 对设计高度定制型项目，仍需要较多样式覆盖
- 如果和 `Taroify` 混用，需严格控制范围，避免两套视觉 token 并存

### 3.7 TDesign 的使用建议

`TDesign` 的微信小程序版本更适合作为“交互和信息架构参考”，而不是本项目的直接主依赖。原因是本项目视觉基调偏“家庭手账 / 传家食谱”，和 TDesign 偏企业级、规整型的默认气质不完全一致。

但它仍然值得参考以下内容：

- 信息密度较高页面的结构组织
- 表单交互规范
- 空状态 / 异常状态设计
- 反馈组件的文案与状态拆分

### 3.8 最终落地建议

建议的组件策略如下：

1. 以 `Taroify` 作为主组件库，承接通用结构组件和交互组件。
2. 业务视觉强的卡片、时间轴、版本差异块、购物清单项、首页手账卡片，全部自定义封装。
3. 如遇 `Taroify` 缺少或表现不稳定的单点组件，再评估是否引入 `@antmjs/vantui` 或局部原生组件。
4. 不建议在项目中同时大面积混用 `Taroify`、`Vant Weapp`、`NutUI` 三套体系。

### 3.9 建议优先使用的组件映射

| 业务场景 | 推荐组件库/实现 | 说明 |
| --- | --- | --- |
| 底部弹层、筛选面板 | `Taroify Popup` / `ActionSheet` | 适合点菜筛选、版本选择、菜谱选择 |
| 日期/多列选择 | `Taroify Picker` | 适合分类、标签、时间筛选场景 |
| 标签、徽标、角标 | `Taroify Tag` / `Badge` | 可快速搭出版本标记、时光数量角标 |
| 标签页切换 | `Taroify Tabs` | 适合详情页“做法/版本/时光轴” |
| 列表骨架屏与空状态 | `Taroify Skeleton` / `Empty` | 首屏和分页加载体验更稳 |
| 轮播与封面切换 | `Taroify Swiper` | 适合菜谱图、时光图集 |
| 时间轴 | `Taroify Timeline` + 自定义卡片 | 组件做骨架，卡片保持项目风格 |
| 固定吸顶/底部操作条 | `Taroify Sticky` + 自定义容器 | 用于详情页与购物清单页 |
| 浮动按钮 | `Taroify FloatingBubble` 或自定义 FAB | 首页、记一笔、分享入口 |
| 表单输入 | 小程序原生 + 自定义样式 | 避免重度依赖库导致样式不可控 |

### 3.10 不建议的选择

- 不建议直接沿用原型 HTML 中的 Tailwind 作为正式实现方案
  - 原型适合表达视觉方向，不适合直接映射到小程序运行时
  - 小程序环境对复杂类名、动态样式和部分 CSS 特性支持不稳定
- 不建议首期引入重型通用 UI 组件库
  - 本项目有强视觉风格，原型明显不是标准后台风
  - 更适合基于设计 token 自建一层业务组件
- 不建议首期做复杂拖拽
  - 小程序拖拽体验和兼容性成本高
  - 点菜台排序先使用“更换餐次 / 上移下移 / 重新选择”

## 4. 关键前置决策

### 4.1 鉴权方案需要在开发前冻结

当前 API 文档默认使用 Cookie Session，但小程序前端与传统 Web Cookie Session 存在天然摩擦，主要问题包括：

- 登录态管理不如 Web 同源场景稳定
- 跨域、环境域名、测试号与正式域名切换时更易出问题
- Taro 请求层与服务端会话续期联调成本高

前端建议：

- 小程序正式接入时，后端新增“面向小程序的 token 鉴权方案”
- 登录成功后返回 `accessToken + refreshToken` 或业务 session token
- 前端通过请求拦截器统一注入 `Authorization`
- Web 管理后台可以继续保留 Cookie Session

如果后端短期必须沿用 Cookie Session，前端仍可先做请求层适配，但需要把登录态作为专项联调风险处理。

### 4.2 原型到小程序的还原边界

`stitch_prd` 中存在大量网页视觉表现，前端实现时应转换为“小程序可稳定落地”的方案：

- `backdrop-filter`、复杂磨砂玻璃效果：保留视觉层级，降级为半透明底色 + 阴影
- 大量绝对定位装饰：保留少量关键装饰，避免影响渲染性能
- 不规则旋转卡片：首页和时光记录中保留轻度角度变化，控制在可读范围
- 长列表大图：首屏压缩 + 懒加载，避免内存峰值过高

### 4.3 设计系统落地方式

基于 `Digital Heirloom` 设计说明，前端需沉淀一套固定 token，而不是在页面里散写颜色值。

建议拆为两类：

- 样式 token：颜色、字体、圆角、阴影、间距、层级
- 业务 token：评分色、版本差异色、购物清单已购态、随机点菜高亮态

## 7. 页面级实现方案

### 7.1 首页

对应原型：`stitch_prd/home`

模块拆分：

- `WeeklyPlanSummaryCard`
- `MomentFeedList`
- `MomentFeedCard`
- `GlobalFab`

实现要点：

- 首屏并行请求：本周菜单摘要 + 最新时光流
- 时光流按时间倒序，卡片支持图片懒加载
- 点击时光卡片时，传 `recipeId + momentId`
- 页面顶部 FAB 统一走“新建菜谱”

### 7.2 菜谱库

对应原型：`stitch_prd/recipe_library`

实现要点：

- 搜索、分类 Tab、标签筛选器为同一筛选状态源
- 列表使用分页加载，不一次性拉全量
- 卡片显示字段直接对齐 API `RecipeListItem`
- 当前版本名称、最近记录日期、版本数、时光记录数都从列表接口返回，避免页面端拼装

### 7.3 菜谱详情

对应原型：`stitch_prd/recipe_details`

Tab 建议：

- 做法
- 版本
- 时光轴

实现要点：

- 详情接口默认返回 `currentVersion`
- “版本”与“时光轴”使用按需请求，不在首屏全量并发
- 详情页承担核心动作入口
  - 记一笔
  - 新建版本
  - 加入计划
- 如果从首页时光流跳入，则默认切到“时光轴”并滚动到对应记录

### 7.4 菜谱新建/编辑

来源：PRD 补齐页，未在原型中单独拆出

字段建议：

- 菜谱名
- 分类
- 标签
- 配料列表
- 步骤列表
- 小贴士

实现策略：

- 创建与编辑复用同一页面
- 保存前在前端做轻校验，至少校验菜谱名
- 创建成功后跳菜谱详情
- 新建成功后可弹出“是否顺手记一笔”

### 7.5 新建版本

对应原型：`stitch_prd/_2`

实现要点：

- 页面进入时拉取“上一版本详情”作为草稿
- 配料、步骤支持增删改
- 修改项在 UI 上有弱高亮，帮助用户建立“版本改良”的心理模型
- 保存成功后回详情页并刷新版本列表

### 7.6 版本对比

对应原型：`stitch_prd/version_comparison_updated_highlights`

实现要点：

- 页面首屏必须优先展示变更摘要
- 详细内容可折叠，不做复杂逐行 diff
- 差异色规范：
  - 新增：绿色系
  - 减少/移除：红棕系
  - 未变：中性色

### 7.7 记一笔

对应原型：`stitch_prd/_1`

实现要点：

- 图片最多 9 张
- 参与人首期使用自由文本标签，不拆独立实体
- 好吃程度和难度使用轻量评分组件
- 支持补记历史日期
- 页面保存成功后返回详情页时光轴 tab

### 7.8 点菜台

对应原型：`stitch_prd/meal_planner`

实现要点：

- 按周展示，顶部日期条支持切周
- 餐次先固定为早餐/午餐/晚餐/加餐
- 添加菜品时弹出“菜谱选择面板”
- 首期不做复杂拖拽，采用编辑按钮 + 替换 / 删除 / 调整餐次
- “生成购物清单”按钮置底固定

### 7.9 随机点菜

对应原型：`stitch_prd/random_pick`

实现建议：

- 阶段 B 实现，但页面与接口、路由、store 先预留
- 由后端返回 session 与结果，前端主要负责筛选面板、动画表现、接受/跳过交互
- 抽取动画不要依赖真实高速切卡，可用伪滚动动画 + 最终结果卡组合实现

### 7.10 购物清单

对应原型：`stitch_prd/shopping_list`、`stitch_prd/_3`

实现要点：

- 按“食材 / 调料”分栏
- 支持勾选、备注、清除已购、复制文本
- 分享图按钮首期可保留入口，能力由后端接口准备度决定
- 更新清单项时采用局部乐观更新

### 7.11 我的 / 分类 / 标签

对应原型：`stitch_prd/my_profile`、`stitch_prd/_7`、`stitch_prd/_6`

实现要点：

- “我的”页只承担入口和简单统计，不放复杂业务逻辑
- 分类管理需要支持排序，但首期可用“上移/下移”
- 标签体系按分组展示
  - 口味
  - 场景
  - 主料

## 9. 状态管理与数据流

### 9.1 状态分层

前端状态分三类：

- 服务端状态
  - 菜谱列表
  - 菜谱详情
  - 版本列表
  - 时光轴
  - 当前周菜单
  - 购物清单详情
- 全局客户端状态
  - 登录态
  - 当前周起始日期
  - 全局筛选草稿
  - feature flag
- 页面本地状态
  - 新建菜谱草稿
  - 新建版本表单
  - 记一笔表单
  - 随机点菜筛选面板展开收起

### 9.2 Query Key 建议

```ts
['session']
['home', 'summary']
['moments', 'latest']
['recipes', 'list', filters]
['recipes', 'detail', recipeId]
['recipes', 'versions', recipeId]
['recipes', 'moments', recipeId, query]
['planner', 'week', weekStartDate]
['shopping', 'detail', shoppingListId]
['categories']
['tags']
['random-pick', 'session', sessionId]
```

### 9.3 本地存储建议

使用 `Taro.setStorageSync` 持久化以下内容：

- 登录 token 或 session 辅助信息
- 最近一次菜谱筛选条件
- 当前周视图定位
- 未完成的表单草稿

不建议持久化：

- 大型列表缓存
- 图片二进制
- 完整详情对象

## 10. 数据模型与前端类型设计

### 10.1 类型来源原则

- 前端类型命名优先对齐 API DTO
- 页面内部再基于 DTO 映射为 ViewModel
- 不直接把接口原始结构散落到模板里使用

### 10.2 核心 ViewModel 建议

- `RecipeCardVM`
- `RecipeDetailVM`
- `RecipeVersionVM`
- `MomentCardVM`
- `WeekPlanVM`
- `ShoppingListVM`
- `RandomPickResultVM`

### 10.3 为什么需要 ViewModel

- 原型中存在大量“面向展示”的组合字段
- 同一个实体在列表和详情的展示重点不同
- 通过 mapper 层可以把“接口变化”和“页面渲染”隔离开

## 11. 接口协作方案

### 11.1 服务层设计

每个领域模块只暴露有限接口：

- `recipeService`
- `momentService`
- `mealPlanService`
- `shoppingService`
- `taxonomyService`
- `randomPickService`
- `mediaService`

服务层职责：

- 拼接路径
- 处理 query 参数
- 解析响应
- 转换异常

### 11.2 请求封装要求

统一请求层需要支持：

- baseURL 管理
- token/Cookie 注入
- 超时处理
- 401 自动跳登录
- 业务错误 toast
- 请求日志埋点

### 11.3 文件上传流程

对齐后端媒体接口，前端建议流程如下：

1. 调 `POST /api/v1/media/upload-token`
2. 使用 `Taro.uploadFile` 上传文件到对象存储
3. 调 `POST /api/v1/media/complete` 登记资源
4. 表单提交时只提交 `mediaId` 或资源列表

### 11.4 联调优先级

建议前端联调顺序：

1. 鉴权与会话
2. 分类 / 标签
3. 菜谱列表 / 创建 / 详情
4. 新建版本 / 版本列表 / 对比摘要
5. 时光记录
6. 点菜台
7. 购物清单
8. 随机点菜

## 12. 视觉系统实现

### 12.1 设计 token 建议

参考 `Digital Heirloom` 设计稿沉淀：

```scss

$color-tertiary: #6e6353;
$color-surface: #fffcf7;

$radius-lg: 16px;
$radius-xl: 24px;

$shadow-card: 0 12px 32px rgba(55, 56, 49, 0.06);
```

### 12.2 字体策略

小程序中不建议强依赖在线字体动态加载，因此建议：

- 标题字：优先系统衬线降级方案或静态资源字体按需引入
- 正文字：系统 sans-serif
- 首期重点保证层级和字重，不把字体作为阻塞项

### 12.3 动效策略

保留原型中的“温暖、轻微、手账感”而不是强动画：

- 页面进入：轻淡入 + 位移
- 卡片：轻缩放反馈
- 随机点菜：单独做一套强调动效
- 长列表页：不做大面积装饰动画

## 13. 小程序端专项约束

### 13.1 可用原生能力

- 图片选择与预览
- 上传
- 震动反馈
- 剪贴板复制
- 分享给好友 / 生成分享图后转发
- 本地缓存

### 13.2 实现约束

- 避免依赖 DOM API
- 避免在滚动容器内叠加多个复杂 fixed 元素
- 避免长列表中使用过多阴影和滤镜
- 图片卡片统一做裁剪比例控制
- 富文本需求首期不引入

### 13.3 页面性能建议

- 首屏接口不超过 2 个关键请求
- 列表页分页，默认每页 10 到 20 条
- 图片统一缩略图地址优先
- 详情页 tab 内容按需加载
- 购物清单更新采用局部刷新

## 14. 错误处理与空状态

每个核心页面必须有三类状态：

- 加载中
- 空状态
- 加载失败

重点空状态：

- 无菜谱
- 无时光记录
- 本周无菜单
- 购物清单尚未生成
- 筛选无结果

重点错误场景：

- 登录失效
- 图片上传失败
- 保存版本失败
- 购物清单生成失败
- 随机点菜候选池为空

## 15. 埋点与可观测性建议

前端建议最少埋点以下事件：

- 创建菜谱

- 创建时光记录
- 加入本周菜单
- 生成购物清单
- 随机点菜开始

同时记录关键页面耗时：

- 首页首屏加载

## 16. 测试策略

### 16.1 页面级自测清单

- 新建菜谱是否自动生成 V1
- 新建版本是否正确复制上一版本
- 时光记录是否最多 9 张图
- 首页跳详情时是否能定位到指定时光记录
- 点菜台是否能绑定具体版本
- 购物清单是否能正确勾选和备注

### 16.2 联调验收清单

- 筛选条件与后端 query 对齐
- 版本切换后详情页是否刷新当前版本
- 最新时光图是否能回填封面
- 购物清单生成结果是否与周菜单一致
- 随机 session 是否能继续重抽/接受

## 18. 关键决策总结

本方案建议坚持以下前端原则：

- 用 Taro React 组织工程，但 UI 能力以微信小程序原生组件和能力为主
- 原型只作为视觉与交互参考，不直接迁移 Tailwind HTML
- 页面按领域分模块，接口类型对齐 API DTO，再映射为 ViewModel
- 随机点菜模块首期可延后，但工程必须提前预留
- 登录态方案必须在编码前冻结，否则联调风险最高
- 复杂拖拽、复杂 diff、复杂玻璃效果都不应成为 MVP 阻塞项
