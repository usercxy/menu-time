# 食光记 MVP 开发任务拆解（个人开发版 Todo 清单）

## Summary

基于你刚确认的范围，这份清单按“最小上线版”拆解，只覆盖最短闭环：

- 做菜谱录入与管理
- 支持版本新增与版本对比摘要
- 支持时光记录
- 支持手动加入本周菜单
- 支持生成购物清单
- 暂不做随机点菜、家庭协作、推荐系统、完整离线同步

建议按“先打通数据闭环，再补页面体验”的顺序推进，目标是每完成一阶段都能看到可运行成果，适合个人开发持续推进。

## 关键接口与默认约定

**核心实体**

- `Recipe`：基础菜谱信息，包含 `id`、`name`、`categoryId`、`tagIds`、`coverImage`、`currentVersionId`、`createdAt`
- `RecipeVersion`：版本内容，包含 `id`、`recipeId`、`versionNumber`、`versionName`、`ingredientsText`、`steps[]`、`tips`、`diffSummary`、`createdAt`
- `Moment`：时光记录，包含 `id`、`recipeId`、`versionId?`、`date`、`content`、`images[]`、`participants[]`、`tasteRating?`、`difficultyRating?`
- `MenuPlan`：本周菜单，包含 `id`、`weekStartDate`、`items[]`
- `ShoppingList`：购物清单，包含 `ingredients[]`、`seasonings[]`
- `Category`、`Tag`：用户自定义分类与标签

**页面最小集**
 
- 首页
- 菜谱库页
- 菜谱新建/编辑页
- 菜谱详情页
- 新建版本页
- 时光记录页
- 点菜台页
- 购物清单页
- 我的页（只放分类/标签管理入口）

**默认实现选择**

- 前端技术栈按“微信小程序原生 + Taro”，考虑跨端
- 后端技术栈按“node + nextjs”
- MVP 仅单用户，不做成员体系
- 版本对比只做“差异摘要”，不做逐行高亮对比
- 点菜只做手动加入，不做随机点菜
- 菜单排序先用“上移/下移”或重新选择，不强依赖拖拽
- 购物清单按关键词规则拆分原料/调料，不做精确用量计算

## Todo 清单

### 阶段 0：需求落地与工程骨架

- [ ] 把 MVP 范围冻结为最小上线版，明确“本期不做”的功能清单
- [ ] 输出一份页面地图，确认页面入口与跳转关系
- [ ] 确认 6 个核心实体的数据字段与关联关系
- [ ] 建立项目基础工程、路由、公共样式、公共工具模块
- [ ] 搭建数据访问层，统一封装增删改查方法，避免页面直接操作底层存储
- [ ] 预留图片上传、日期处理、ID 生成、差异摘要生成的工具函数入口

**完成标准**

- 项目可启动
- 页面路由能跳转
- 数据结构固定下来，后续页面不再反复改字段

### 阶段 1：菜谱基础闭环

- [ ] 实现菜谱列表页：展示封面、菜名、当前版本、最近记录日期
- [ ] 实现分类筛选、标签筛选、菜名搜索
- [ ] 实现菜谱新建页：名称、分类、标签、主料、步骤、小贴士
- [ ] 新建菜谱时自动创建 `V1` 版本
- [ ] 实现菜谱编辑能力，允许修改基础信息
- [ ] 实现分类管理页：新增、编辑、删除、排序
- [ ] 实现标签管理页：新增、编辑、删除

**完成标准**

- 用户可以从 0 创建菜谱并在列表中看到
- 分类、标签、搜索能正常筛选
- 菜谱基础字段能稳定保存和回显

### 阶段 2：版本管理

- [ ] 在菜谱详情页展示当前版本内容
- [ ] 增加“新建版本”入口
- [ ] 新建版本时自动复制上一版本内容作为初始值
- [ ] 保存新版本时自动递增版本号
- [ ] 生成差异摘要，至少覆盖主料变化、标签变化、步骤数变化
- [ ] 在详情页展示版本列表
- [ ] 点击历史版本可查看该版本内容与差异摘要
- [ ] 允许将某个历史版本切换为当前版本，或明确只允许最新版本为当前版本并在界面说明

**完成标准**

- 同一道菜可以有多个版本
- 当前版本与历史版本关系清晰
- 版本新增后不会覆盖旧内容
- 用户能直观看到“这次改了什么”

### 阶段 3：时光记录

- [ ] 在菜谱详情页增加“记一笔”入口
- [ ] 实现时光记录表单：日期、内容、图片、参与人、评分
- [ ] 保存记录时默认绑定当前菜谱，可选绑定当前版本
- [ ] 在保存新版本后增加“是否顺手记录本次做菜时刻”的承接流程
- [ ] 在菜谱详情页展示按时间倒序的时光轴
- [ ] 时光记录卡片展示日期、内容摘要、图片缩略图、评分
- [ ] 规则实现：最新记录图片可回填为菜谱封面，若用户未自定义封面

**完成标准**

- 用户可以为菜谱记录做菜时刻
- 同一道菜能积累多条时间线记录
- 菜谱详情页能同时看到“做法”和“回忆”

### 阶段 4：点菜台

- [ ] 建立“本周菜单”数据结构，按周一到周日存储
- [ ] 实现点菜台周视图
- [ ] 点击某天卡槽可从菜谱库选择菜品
- [ ] 支持选择具体版本加入菜单
- [ ] 支持移除已加入的菜
- [ ] 支持简单排序调整，优先用低成本交互实现
- [ ] 首页展示“本周已规划 X 道菜”并跳转点菜台

**完成标准**

- 用户能把本周要做的菜排进日历
- 菜品与版本的对应关系不丢失
- 首页和点菜台之间形成清晰导航

### 阶段 5：购物清单

- [ ] 从本周菜单解析所有已选菜品
- [ ] 按关键词规则拆分“原料/菜品”和“调料”
- [ ] 对相同名称进行去重合并
- [ ] 生成购物清单页，分栏展示菜品和调料
- [ ] 每条清单支持勾选“已购”
- [ ] 支持填写数量备注
- [ ] 支持复制文本导出
- [ ] 如时间够，再补“生成图片分享”；不够则延后到上线后迭代

**完成标准**

- 本周菜单能一键转成可用购物清单
- 用户能边买边勾选
- 清单导出至少支持文本复制

### 阶段 6：首页与导航收口

- [ ] 首页展示本周菜单快捷入口
- [ ] 首页展示最近时光记录流
- [ ] 最近记录点击后可直达对应菜谱详情
- [ ] 底部导航与页面返回路径整理完整
- [ ] 空状态补齐：无菜谱、无记录、无菜单、无购物清单

**完成标准**

- 用户第一次进入不会迷路
- 首页具备“看最近动态 + 继续规划本周菜单”的作用
- 所有核心页面都能从主路径访问

### 阶段 7：体验收尾与上线前检查

- [ ] 表单校验补齐，至少校验菜谱名称必填
- [ ] 删除操作增加二次确认
- [ ] 列表、详情、保存、删除后的提示文案补齐
- [ ] 图片数量限制、文本长度限制、日期回填逻辑补齐
- [ ] 异常场景处理：无数据、加载失败、保存失败
- [ ] 检查页面性能，避免列表卡顿和重复请求
- [ ] 整理埋点或最小日志，方便观察使用情况
- [ ] 准备一套演示数据，方便自测和上线展示

**完成标准**

- 核心流程没有明显阻塞点
- 页面有基本容错能力
- 能拿给真实用户试用而不至于频繁报错

## Test Plan

**核心流程测试**

- 创建第一道菜谱后，能看到 `V1` 自动生成
- 编辑菜谱基础信息后，列表和详情回显一致
- 新建 `V2` 后，`V1` 内容仍保留且差异摘要正确
- 同一菜谱下创建多条时光记录，详情页按时间倒序显示
- 时光记录绑定版本后，详情页能看到关联信息
- 点菜台可为不同日期加入不同菜谱版本
- 从点菜台生成购物清单后，原料和调料能正确拆分并去重
- 勾选已购、填写备注、复制文本后状态正确保存

**边界场景测试**

- 无分类、无标签时也能创建菜谱
- 主料为空、步骤为空时仍能保存菜谱
- 同一道菜连续创建多个版本时版本号不冲突
- 同名原料在多个菜里重复出现时只保留一项
- 菜单为空时生成购物清单要有清晰提示
- 删除菜谱后，相关版本、时光记录、菜单引用要有明确处理策略

**验收标准**

- 用户能在 10 分钟内完成“建菜谱 → 记一次做法 → 加入本周菜单 → 生成购物清单”的完整闭环
- 不依赖随机点菜，也能形成稳定可用的 MVP
- 数据模型可自然扩展到后续“随机点菜”和“家庭协作”

## Assumptions

- 本轮 Todo 只服务个人开发推进，不拆到工时级别的设计稿任务
- 随机点菜整体延后到下一个版本，不混入当前开发主线
- 完整离线同步、多人权限、推荐系统、打印导出均不纳入本期
- 如果开发过程中某个交互实现成本过高，优先保留业务能力，弱化动效和复杂交互
